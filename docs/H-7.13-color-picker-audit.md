# H-7.13 — Simplified Social Animator color picker (audit)

**Branch:** `phase-h7-13-color-picker-simplification` (cut from `c8ea10f` / v1.38)
**Author:** Audit gate — implementation pauses on user sign-off.

**Premise.** Three of the four tools — Listing Flyer, Open House Promo, Listing
Presentation — share an "elegant, simple, easy to understand" color picker
pattern. Social Animator's per-template inline color UI feels overwhelming by
comparison (56 color pickers across 10 templates with no brand inheritance).
This phase brings Social Animator into line with the other three tools'
pattern, exactly. No new pattern invented; nothing Social-Animator-flavored.

---

## 1. The reference pattern (three tools, identical)

`FlyerForm.tsx`, `PromoForm.tsx`, and `PresentationForm.tsx` implement an
identical color picker pattern. Differences between them are limited to the
section copy ("this flyer" vs. "this promo" vs. "this presentation") and
OH Promo's use of the `effectiveBrandAccent()` helper (auto-derives a darker
accent from primary when the brand profile's accent is empty or legacy white).
The structural pattern is the same in all three.

### 1.1 How many pickers
**3 pickers per tool:** Primary, Accent, Background.

### 1.2 Which colors are exposed
| Label | Role |
| --- | --- |
| Primary | Status/badge fills, hero color, accent strokes |
| Accent | Secondary highlights — small text accents, sometimes a derived auto-default (OH Promo) |
| Background | Page/output fill |

### 1.3 Brand inheritance mechanism (literal, copy-paste from the three forms)
```ts
const effectivePrimary = draft.primaryColor || brand.primaryColor;
const effectiveAccent = draft.accentColor || brand.accentColor;   // (Flyer + Presentation)
const effectiveAccent = draft.accentColor || effectiveBrandAccent(brand);  // (OH Promo only)
const effectiveBackground = draft.backgroundColor || brand.backgroundColor || "#ffffff";
```
- **Storage:** empty string in the draft = "use brand fallback"; any non-empty
  hex = user override.
- **Render:** the `effectiveX` value is what's passed to the renderer/preview.
- **Mutation:** typing a hex in the input writes the override into `draft.X`.

### 1.4 UI placement
- **Top of the form**, above every content section. Listing Flyer puts it as
  the very first block; OH Promo + Listing Presentation place it as the first
  named block in a sectioned form ("BRAND COLORS" section before "EVENT" /
  "PROPERTY" / etc).
- The three pickers sit in a `grid-cols-2 sm:grid-cols-3` row — wraps to 2
  rows on mobile portrait, single row from sm+.
- Section label: `"Brand colors (this {flyer|promo|presentation})"` in the
  uppercase 10-px tracking style.
- Helper text below the row:
  `"Override colors for this {flyer|promo|presentation} only — your brand profile in Settings is unchanged."`

### 1.5 "Reset to brand defaults" affordance
- Top-right corner of the section header, opposite the "Brand colors …" label.
- **Visible only when an override exists** (`hasColorOverride = !!draft.primaryColor || !!draft.accentColor || !!draft.backgroundColor`).
- Click clears all three draft color fields to empty string — the
  `effectiveX = draft.X || brand.X` resolution then picks up the live brand
  values automatically.
- Style: bare text button `text-[10px] text-neutral-500 hover:text-[#4ef2d9]`.
- Glyph + label: `"↺ Reset to brand defaults"`.

### 1.6 ColorInput component
All three tools use a locally-declared `ColorInput` component with an
identical signature: a label above (uppercase 9px tracking), a 44×44pt swatch
acting as the native `<input type="color">` host, and a hex text input
sharing the row. No global component — each form file declares its own
near-identical copy (acceptable duplication: the picker is the form's main
visual hook, and inlining keeps the file self-contained).

### 1.7 Summary of the reference contract
| Aspect | Pattern |
| --- | --- |
| Picker count | 3 per tool (Primary / Accent / Background) |
| Placement | Top of form, above content |
| Brand inheritance | `draft.X \|\| brand.X` resolved at render time |
| Storage | Empty draft string = brand fallback; non-empty = override |
| Reset button | Top-right of section, visible only when override exists |
| Helper text | "Override colors for this X only — your brand profile in Settings is unchanged." |
| Section copy | "Brand colors (this {flyer|promo|presentation})" |
| Per-template anomaly | OH Promo uses `effectiveBrandAccent(brand)` to auto-derive accent when blank |

## 3. Proposed target

### 3.1 Architecture (matches reference pattern exactly)

For each Social Animator template:
- **3 color pickers per template (Primary / Accent / Background).** Exceptions
  documented in §4.
- **Rendered as a top-of-form section** labeled `"Brand colors (this template)"`,
  before the existing content fields. Mirror the reference tools' placement
  pixel-for-pixel.
- **Brand inheritance resolved at render time** in `TemplateEditor`'s
  `buildState` `useMemo`:
  ```ts
  const effectivePrimary = state.primary || brandSettings.primaryColor;
  const effectiveAccent = state.accent || effectiveBrandAccent(brandSettings);
  const effectiveBackground = state.background || brandSettings.backgroundColor || templateBg;
  ```
  The `templateBg` fallback per template is whatever the template's
  current `background` FieldDef default was (e.g. `"#000000"` for most,
  `"#0a0a0a"` for listing-showcase) — preserves the template's visual identity
  when no brand background is set.
- **Storage:** empty string in `state.primary` = "use brand fallback"; non-empty
  hex = override. Same as the reference tools.
- **"↺ Reset to brand defaults" affordance** at the section's top-right.
  Replaces today's misleadingly-named "Reset colors to brand defaults" button
  in the toolbar sidebar (which will be removed — it was always lying).
- **Helper text below the row:**
  `"Override colors for this template only — your brand profile in Settings is unchanged."`

### 3.2 BrandProfileForm changes
**None.** Already at primary + accent + agent info per §2.2. No commit needed.

### 3.3 No new brand color UI inside any template editor
- No "Brand colors" palette section like the reverted H-7.12 D
- No "Customize colors" accordion
- No collapsible per-template overrides UI
- The 3 (or 3+exception) pickers ARE the entire color UI per template, placed
  inline at the top of the form, identical to the three reference tools.

### 3.4 Mapping helper functions inside each build()
Where the current build() reads 6+ named color fields, the new build()
internally re-derives them from the 3 resolved colors via small local
helpers. For example, [listing-card.ts:167-179](src/templates/listing-card.ts#L167-L179)
currently reads `state.statusColor`, `state.statusTextColor` for the badge
fill + text. The new build reads `effectivePrimary` + `effectiveBackground`,
and the badge fill = primary while badge text = a contrast-safe hardcoded
value (e.g. `"#0a0a0a"` against light primaries, `"#ffffff"` against dark —
or just hardcode one — see §4 per-template).

This collapse is the actual surgery. It is concentrated in
the 10 build() functions, not in the picker UI.

## 7. Open questions for sign-off

I will pause on this audit and await explicit approval (or override) on each
of the following before running Commit 2. Bullet-point answers welcome.

### Q1. Section copy
Reference tools say "Brand colors (this {flyer|promo|presentation})". For
Social Animator templates, the most literal match is **"Brand colors (this
template)"**. Acceptable? Or do you want a per-template noun ("Brand colors
(this carousel)", "Brand colors (this card)", etc.)?

### Q2. Accent fallback mechanism
The three reference tools split:
- Listing Flyer + Listing Presentation: `draft.accentColor || brand.accentColor`
- OH Promo: `draft.accentColor || effectiveBrandAccent(brand)` (auto-derive
  from primary when blank or legacy white)

OH Promo's `effectiveBrandAccent()` is the better default — it never lands
on white-on-white. **Default proposal: use `effectiveBrandAccent(brand)` in
the Social Animator resolution** so blank accent always lands on a usable
derived color. Confirm or override.

### Q3. Background fallback per-template
Each template currently has a hardcoded background default — e.g.
`"#000000"` for most, `"#0a0a0a"` for listing-showcase. The proposed
fallback chain is:
```
state.background || brand.backgroundColor || templateDefaultBg
```
where `templateDefaultBg` is each template's existing default value. This
preserves visual identity per template when no brand background is set.
Acceptable? Or do you want a uniform `#000000` floor for everything?

### Q4. qa-card exception
Proposed: surface "Question panel" as a 4th picker on qa-card, with default
`"#4bc9f0"` (sky blue). Rationale: question + answer panels are visually
distinct by design; collapsing both into Accent erases the contrast.
Confirm exception is warranted, or instruct me to collapse anyway.

### Q5. The 22 "drop entirely" hardcodes
Twenty-two color fields collapse to hardcoded values in the new build() —
mostly `#ffffff` (white text), `#9ca3af` (muted text), `#0a0a0a` (near-black).
A user who previously typed in a custom hex for, say, `statsColor` would
lose that override on this migration. Migration semantics:
- **Default proposal:** when a user opens a template after the migration,
  the new 3-picker state initializes empty (brand fallback). Old localStorage
  values under the dropped field keys are silently ignored (the renderer no
  longer reads them, but the localStorage key may persist as dead data until
  a future cleanup).
- Should the migration actively clear the dead keys from
  `socanim_colors_<template-id>` localStorage? Or leave them as no-op dead
  data?

### Q6. Touching the existing toolbar "Reset colors to brand defaults" button
Today's [TemplateEditor.tsx:384-389](src/components/TemplateEditor.tsx#L384-L389)
button sits in the sticky toolbar alongside "Replay animation". It will be
removed — replaced by the section-local "↺ Reset to brand defaults"
affordance inside the new top section (matching the three reference tools).
Confirm OK to remove, or keep both?

### Q7. Commit batching
Plan above splits into 5 commits (audit + scaffolding + 3 template batches).
Acceptable, or would you prefer:
- A single combined commit covering scaffolding + all 10 templates?
- One commit per template (~11 commits total)?
- Or this 5-commit plan as written?

### Q8. Visual regression risk on listing-showcase
listing-showcase has 14 named color reads and 11 → 3 collapse. The highest
risk is that one of the "drop entirely" hardcodes (white on white, or muted
on muted) produces an unreadable text element on a brand background the
agent later picks. Mitigations available:
- (a) Choose hardcodes that work on dark AND light brand backgrounds (the
  template's current default `#0a0a0a` is dark — anything on top is white).
  But if a user sets a white brand background, addressColor=white becomes
  unreadable.
- (b) Add a luminance check that flips white ↔ near-black based on the
  effective background — adds a helper but is the robust fix.
- (c) Punt — assume agents pick dark backgrounds matching the template's
  current default, and document the limitation.

Default proposal: **(a) for this phase, (b) as a follow-up if the smoke
test surfaces a real case.** Confirm or override.

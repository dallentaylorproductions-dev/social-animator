# v1.47 / A1 — Seller Presentation: integration audit

Document-only audit: map the locked Seller Presentation contract (substrate
v1.0 §§2, 3, 3.3, 3.4, 4, 8, 9) onto the SEP codebase as it exists today,
produce a slot/migration table, and propose a commit plan for A2+.

No feature code in this pass. Every file path and symbol below was opened
during the audit; nothing is inferred from a guess.

## 0. TL;DR

The codebase is closer to substrate v1.0 than the packet's caution implies:

- The **wizard shell** (5-step + StepIndicator + prev/next + SSR-safe load) is
  already proven twice over in [src/app/seller-intelligence-report/page.tsx](src/app/seller-intelligence-report/page.tsx) and
  [src/app/open-house-prep/page.tsx](src/app/open-house-prep/page.tsx). Reusable verbatim as a layout pattern.
- The **dual-output pattern** (private prep PDF + public web URL) is already
  shipping for OH Prep — [src/lib/share-urls.ts](src/lib/share-urls.ts), [src/app/h/[slug]/page.tsx](src/app/h/%5Bslug%5D/page.tsx), the
  [/api/oh-prep/publish](src/app/api/oh-prep/publish/route.ts) + [/api/oh-prep/revoke](src/app/api/oh-prep/revoke/route.ts) endpoint pair,
  [src/tools/open-house-prep/output/handout-page.tsx](src/tools/open-house-prep/output/handout-page.tsx), and the dynamic OG image at
  [src/app/api/og/[slug]/route.tsx](src/app/api/og/%5Bslug%5D/route.tsx).
- The **Property primitive** exists in usable form as `useListingProfile`
  ([src/lib/listing-profile.ts](src/lib/listing-profile.ts)).
- The **Agent Profile primitive** exists as `useBrandSettings`
  ([src/lib/brand.ts](src/lib/brand.ts)) with explicit SSR-safe load that matches substrate §9.7
  verbatim (see [src/lib/brand.ts:203-216](src/lib/brand.ts#L203-L216) — the comment even names React error #418).
- The **Comp shape** already carries `source` + `fieldConfidence`
  ([src/tools/seller-intelligence-report/engine/types.ts:33-50](src/tools/seller-intelligence-report/engine/types.ts#L33-L50)),
  ready for the Lane C photo-to-comp plug-point with no edit.
- The **HandoutType discriminator** is open-ended (`(string & {})`) by design
  ([src/lib/share-urls.ts:38-41](src/lib/share-urls.ts#L38-L41)) — adding a new
  `'seller-presentation'` type is a single dispatch arm at each consumer.
- The **dashboard auto-buckets new skills** via the required `category` field
  with no edits to `AllSkillsSection`; chains derive from `recommendedNextSkills`.
  Root-cause fix from the v1.44.1 dropout bug, intact.

What's **genuinely new** and must be built (not reused):
- Thin Client primitive (substrate §2 — no analogue exists)
- Stable IDs across primitives (substrate §2.3 — zero hits across the tree)
- Converged draft shape keyed by `workflowInstanceId` (substrate §9.5)
- `SkillStatus` orchestration-readable record (substrate §3.3)
- `availability` field on CallableSkill + the three-dimensional resolver (§3, §8)
- Public-payload serializer + its allowlist test (§4) — see Risk R-1 below
- The Seller Presentation skill itself: 5-step wizard, two renderers, publish API

One **substrate-contradicting pattern** that today's OH Prep silently relies on,
and that the new Seller Presentation **must not replicate** — see Risk R-1.

Two structural observations:

**(a) `useBrandSettings` already implements the §9.7 SSR-safe pattern.** The hook
initializes to `DEFAULT_BRAND` on both server and client first-render and only
reads `localStorage` inside `useEffect`. The inline comment at
[src/lib/brand.ts:203-209](src/lib/brand.ts#L203-L209) explicitly cites React
error #418 and the v1.45.1 hydration regression — substrate alignment is verbatim.
`useListingProfile` follows the same shape ([src/lib/listing-profile.ts:108-148](src/lib/listing-profile.ts#L108-L148)).
Both SIR + OH Prep wizards mirror it ([src/app/seller-intelligence-report/page.tsx:53-69](src/app/seller-intelligence-report/page.tsx#L53-L69),
[src/app/open-house-prep/page.tsx:63-86](src/app/open-house-prep/page.tsx#L63-L86)).

**(b) `useListingProfile` is single-tenant.** One active listing per browser; no
`propertyId`. For the SP to satisfy §2.3 (every primitive instance gets a local
UUID on first write), the cleanest move is to assign `propertyId` opportunistically
on the next write inside `useListingProfile`'s save path, not to bulk-migrate.
The `ListingProfile` shape can carry `propertyId?: string` without disturbing
any existing consumer; downstream the SP simply requires it.

## 8. Risks, contradictions, and decisions Dallen owes

### Risk R-1 — OH Prep publishes the full draft to KV (substrate §4 violation)

**Finding.** [src/app/api/oh-prep/publish/route.ts:68-72](src/app/api/oh-prep/publish/route.ts#L68-L72)
spreads the clamped draft into `data`:
```ts
const data: Record<string, unknown> = {
  ...draft,
  agentContact: payload.agentContact ?? { email },
};
```
The OH Prep draft contains `preEventNotes` (explicitly labelled
"never appears on visitor handout" at [src/tools/open-house-prep/engine/types.ts:72-73](src/tools/open-house-prep/engine/types.ts#L72-L73))
and `followUpCommitments` (post-event agent-only). Today the rendered HTML
omits them ([src/tools/open-house-prep/output/handout-page.tsx](src/tools/open-house-prep/output/handout-page.tsx)
doesn't read them), but the values **sit in the public KV record**. If a future
endpoint exposes the JSON record (or KV is breached), private prep notes leak.

**Implication for SP.** The Seller Presentation must **not** replicate this
pattern. The §5.5 serializer + §5.7 publish-route design closes this gap for SP
by construction (the route calls `toPublicPayload` and passes its result to
`publishHandout`; the draft never enters KV).

**Implication for OH Prep.** Substrate-non-compliant. Fix is straightforward
(write an OH Prep `toPublicPayload` mirroring SP's) but **out of scope for this
lane**. Feeds substrate v1.1 / a v1.48 cleanup commit. Calling it out here so
the next audit pass doesn't miss it.

### Risk R-2 — `useListingProfile` is single-tenant; SP workflow requires `propertyId` to be stable across instances

**Finding.** No `propertyId` exists on `ListingProfile` ([src/lib/listing-profile.ts:28-37](src/lib/listing-profile.ts#L28-L37)).
The substrate's converged `WorkflowInstance` needs `resolvedPrimitives.propertyId`
to be a stable handle so resume-after-reload works, and so two SP instances on
the same property can recognize each other.

**Resolution.** Add `propertyId?: string` to `ListingProfile` (additive; no consumer
breaks) and assign on next save when missing. The first SP write creates it; subsequent
SPs for the same browser-active listing inherit it. **Multi-listing support is out
of scope this lane** — see Open question Q-1.

### Risk R-3 — `recommendedNextSkills` points at a skill that may sunset

**Finding.** The current SIR record has `recommendedNextSkills: ['listing-presentation']`
([src/tools/seller-intelligence-report/skill.ts:52](src/tools/seller-intelligence-report/skill.ts#L52))
— pointing at the legacy one-pager skill. If/when SP supersedes that one-pager,
the chain rots. **Not blocking** because SP introduces its own `recommendedNextSkills`
list and the existing pointer is benign until someone decides to delete the legacy
skill. Flag for substrate v1.1.

### Risk R-4 — `tools/index.ts` is a parallel orphaned registry

**Finding.** [src/tools/index.ts](src/tools/index.ts) maintains a separate `TOOLS`
list (Social Animator + Listing Flyer + Listing Presentation + Open House Promo)
that has **no consumer** in `src/app` or `src/components`. SIR and OH Prep aren't
in it. The Seller Presentation does not need an entry there. Worth deleting in
a future cleanup; not blocking.

### Open question Q-1 — Client primitive: single-active or multi-client?

Substrate §2 calls the Client "thin" + "optional" — read as "name a recipient
and personalize the output". The cheapest v1 implementation is single-active
(matches `useBrandSettings`/`useListingProfile`'s shape). A list-of-clients
hook (`useClients()` returning `Client[]` + an active-client selector)
is materially more code and only pays off when an agent juggles multiple
sellers simultaneously.

**Recommendation.** Single-active for v1.47. The `clientId` is stable
(`generateId('client')` on first save) so a later migration to multi-client
keyed by `clientId` is non-breaking. Confirm Y/N.

### Open question Q-2 — Tier model for `availability` resolver

The codebase has one subscription bit today (`hasActiveSubscription` —
[src/lib/subscription.ts](src/lib/subscription.ts)). Substrate §3/§8 implies a
two-tier model (Base + Pro). The audit's plan is to wire `resolveAvailability`
against the existing single-bit subscription **for now**, treating an active
sub as Pro. A real Base vs. Pro split needs a Stripe price-ID check or a tier
flag in the user record.

**Recommendation.** Stub `tierFor(email): 'base' | 'pro'` returning `'pro'`
when `hasActiveSubscription` is true, else `'base'`. Leave a `// TODO: tier
model` comment. Confirm the v1.47 wedge accepts the stub.

### Open question Q-3 — Dashboard workflow card: retarget Seller Win or add SP card?

Two viable wirings:

**Option A (recommended).** Retarget `seller-win` workflow's
`primarySkillId` from `'seller-intelligence-report'` to `'seller-presentation'`,
keep SIR as a chained recommendation. Reads as "Seller Win System now leads with
the new presentation; the old prep doc is the secondary action." Simplest
dashboard surface.

**Option B.** Add a separate `seller-presentation` workflow card alongside `seller-win`.
Two cards compete for the same agent attention; the priority order in
[src/app/dashboard/workflows.ts:94-101](src/app/dashboard/workflows.ts#L94-L101)
already has heuristics that can be extended.

**Recommendation.** A. Confirm.

### Open question Q-4 — Premium theme catalog: how many themes, where do they live?

Substrate §3.4 says `premiumThemes: 'pro'` (the upsell). The audit doesn't
locate any existing theme abstraction in the SEP codebase. The minimum viable
catalog is:

- One **default theme** (Base): the mobile-first layout from
  `presentation-page.tsx`, brand colors from `BrandSettings`.
- One or two **premium themes** (Pro): variations in typography / hero
  treatment / motion.

**Recommendation.** Ship A2 with one default + one premium. Themes live at
`src/tools/seller-presentation/themes/{default,editorial}.tsx`, each exporting
a React component over the same `PublicPayload` prop shape. Add the catalog
in a dedicated commit so it's revertable. Confirm count + premium-theme names.

### Open question Q-5 — `emitsEvents` declaration without an emitter

The packet says "Seller Presentation may declare emitsEvents but the Event
endpoint itself is not built here". Two readings:

(a) Declare on the skill record, no runtime emission (no-op). Honest declaration.

(b) Declare AND emit synchronously to a future `/api/events` that returns 204
when not yet built. Risks a 404 storm.

**Recommendation.** (a). The declaration is metadata; emission is a separate
commit. Confirm.

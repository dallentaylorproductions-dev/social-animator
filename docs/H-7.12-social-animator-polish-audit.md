## A. `objectList` FieldDef type

### State-storage strategy: JSON-string in `TemplateState`

`TemplateState = Record<string, string>` today. The `stringList` precedent (H-7.7.1) stored arrays as newline-joined strings under a single key вҖ” preserved the flat-map shape, kept `getDefaultState` and `loadSavedColors` unchanged. `objectList` should follow the same pattern with **JSON-string** under one key.

Per-render cost: `JSON.parse(state[key])` once per `build()`. Negligible.

The alternative ("synthesized keys" like `images.0.imageUrl`) would require teaching `getDefaultState`, the color-save helpers, and TemplateEditor's render loop about variable-arity keys. Wider blast radius for no UX gain.

### Proposed TypeScript shape

```ts
// New: per-item field spec inside an objectList schema.
export type ObjectListItemType =
  | "text"
  | "textarea"
  | "image"
  | "number"
  | "currency"
  | "phone";

export interface ObjectListItemSpec {
  type: ObjectListItemType;
  label: string;
  placeholder?: string;
  max?: number;            // char cap for text/textarea, value cap for number
}

export interface ObjectListSchema {
  [fieldName: string]: ObjectListItemSpec;
}

// FieldType union gains "objectList"
export type FieldType =
  | "text" | "textarea" | "color" | "image" | "select"
  | "stringList" | "currency" | "number" | "phone"
  | "objectList";        // NEW

// FieldDef gains an optional `schema` (objectList-only).
// `max` already exists from stringList; reused here as the list-length cap.
export interface FieldDef {
  // ... existing fields unchanged
  schema?: ObjectListSchema;     // NEW вҖ” required when type === "objectList"
}
```

### Image-asset handling for objectList items

Image fields nested inside objectList items can't live inside `TemplateAssets = Record<string, HTMLImageElement | null>` as-is. Two options:

1. **Inline data URLs inside the JSON** вҖ” image stored as data URL string within the object's `imageUrl` field. `build()` decodes data URLs to `HTMLImageElement` itself (via `new Image()`), or the editor synthesizes assets keys.
2. **Synthesized asset keys** вҖ” when materializing, the editor pushes `assets["images.0.imageUrl"]`, `assets["images.1.imageUrl"]`, etc. into the existing flat `TemplateAssets` map. `build()` reads them at synthesized keys.

**Recommend option 2** (synthesized keys). `build()` can't synchronously decode data URLs into `HTMLImageElement` without async work; the existing pattern already pre-materializes images in the editor. The synthesized key convention is consistent with the existing pattern and keeps `build()` synchronous.

Convention: `${fieldKey}.${index}.${imageFieldName}` (e.g. `images.0.imageUrl`).

### Editor renderer

Stacked-card UI per objectList item:
- Each item rendered as a collapsible card with the schema's fields as nested form inputs (re-using existing renderers: TextInput, ImageField, CurrencyInput, NumberInput, etc.)
- Header per card: "Photo 1", "Photo 2", вҖҰ (or schema-driven label)
- Г— button to remove (when length > 0); + button to add (when length < max)
- "(n / max)" counter in the field's outer label, mirroring `stringList`

State updates:
- Edit an item field вҶ’ JSON.parse вҶ’ mutate index вҶ’ JSON.stringify вҶ’ updateField(key, newJson)
- Add вҶ’ push empty object filled with defaults derived from schema вҶ’ JSON.stringify вҶ’ updateField
- Remove вҶ’ splice at index вҶ’ JSON.stringify вҶ’ updateField

Add/remove also need to update the synthesized image-asset keys (push to / splice out of `assets`).

### Risk + scope

Pure addition. No template uses `objectList` after this commit. Build must pass with zero callsite changes.

**Estimated lines:** ~120 in `TemplateEditor.tsx` (renderer block + state helpers) + ~30 in `types.ts` (type definitions). **Single commit.**

### Proposed shape

```ts
fields: [
  { key: "title",       ... unchanged },
  { key: "titleColor",  ... unchanged },
  { key: "subtitle",    ... unchanged },
  { key: "subtitleColor", ... unchanged },
  {

    label: "Photos",
```

`build()` updates to:

```ts
const images = JSON.parse(state.images || "[]") as Array<{imageUrl: string; caption: string}>;
const photos: HTMLImageElement[] = [];
images.forEach((_, i) => {
  const img = assets?.[`images.${i}.imageUrl`];
  if (img && img.complete && img.naturalWidth > 0) photos.push(img);
});
```

### вҡ  Decision needed: caption rendering

The brief's schema includes a `caption` field per image, BUT today's `listing-carousel.ts` render loop does **not** draw any per-photo caption вҖ” only a single global title/subtitle. The brief also says "DO NOT touchвҖҰ visual choreography, animation timing, MP4 render pipeline вҖ” pure form/editor refactor, no visual changes intended."

These are in tension. Two interpretations:

- **B.1 (data-only):** add the `caption` field to the schema, store captions, but DON'T render them in `build()`. Captions become invisible data вҖ” captured for a future visual update. **Compatible with "no visual changes" rule.**
- **B.2 (data + visual):** add the caption field AND extend `build()` to draw each caption below or beneath the centered photo. **Contradicts "no visual changes" rule.**

**Recommend B.1** to honor the discipline rule. We can ship B.2 separately as a small visual phase once you've approved the layout (font, position, max-width, animation timing). Speak up if you intended B.2.

### Other notes

- Item cap changes 6 вҶ’ 8 (per brief). That's a feature expansion (more carousel photos allowed). Brief intentional.
- `sampleAssets` block (lines 18-25) seeds photo1..photo6 with sample images for the picker preview. After the rewire, `sampleAssets` needs to use the synthesized-key convention: `"images.0.imageUrl"`, `"images.1.imageUrl"`, etc. Schema-driven preview seeding will need a similar treatment.

## C. `useListingProfile` hook + property data dedup

### Templates carrying duplicated property fields

Only **two** Social Animator templates currently carry listing/property data:

| Template | Property fields |
|---|---|
| `listing-card` | heroPhoto, status, price, address, cityState, beds, baths, sqft (8) |
| `listing-showcase` | heroPhoto, status, address, cityState, price, beds, baths, sqft (8, plus features which is template-specific) |

The other 8 templates (carousel, qa-card, before-after, numbered-process, market-update, stat-highlight, grid-comparison, testimonial-card) hold no listing data вҖ” their content is non-property-specific (process steps, stats, quotes, before/after pairs, etc.).

So sub-phase C affects exactly **2 templates** with **7 shared fields** between them.

### Proposed `ListingProfile` schema

```ts
export interface ListingProfile {
  heroPhoto: string;       // data URL
  status: string;          // "Just Listed" | "Just Sold" | "Open House" | ...
  address: string;
  cityState: string;
  price: string;           // formatted display, e.g. "$685,000" (matches H-7.10 contract)
  beds: string;
  baths: string;
  sqft: string;            // formatted display, e.g. "2,840"
}

const DEFAULT_LISTING_PROFILE: ListingProfile = {
  heroPhoto: "",
  status: "Just Listed",
  address: "",
  cityState: "",
  price: "",
  beds: "",
  baths: "",
  sqft: "",
};
```

### Hook contract (mirrors `useBrandSettings`)

```ts
export function useListingProfile(): {
  settings: ListingProfile;
  heroPhotoImg: HTMLImageElement | null;
  update: (next: Partial<ListingProfile>) => void;
};
```

- Hydrates from `localStorage` key `socanim_listing_profile` on mount (SSR-safe via the existing pattern)
- `update()` writes to localStorage on every change
- Materializes `heroPhotoImg` from the stored data URL (same pattern as `logoImg` in `useBrandSettings`)

### Editor integration

**Two questions to settle before implementing:**

1. **One profile or multiple?** Brand profile is a single profile (one set of agent info). Should listing profile be the same (one current listing) or named profiles ("1247 Maple", "1100 Cedar", etc.)?
   - **Recommend single profile.** Mirrors the brand-profile pattern, simpler UX, easier to ship. Agents typically work on one listing at a time. Named-profile UX is bigger scope and can land later if needed.

2. **Auto-save or explicit "Save as listing profile" button?**
   - **Recommend auto-save** on every form edit. Matches brand profile, no separate UX. Agents who don't want auto-population can manually clear it.

### Per-template integration

In `TemplateEditor`, listing-card and listing-showcase get a special-case (similar to how listing-showcase already special-cases brand-profile injection for agent fields, see [TemplateEditor.tsx:163-193](src/components/TemplateEditor.tsx#L163-L193)):

- When the template is listing-card / listing-showcase AND the corresponding state field is empty, fall back to the listing profile value as the default
- Editing the form field auto-saves back to the listing profile
- Per-template overrides remain supported (user can leave the form field set to a one-off value; profile won't be updated unless they explicitly... hmm вҖ” see below)

вҡ  **Edge case to resolve:** if editing always writes back to the profile, there's no way for the user to make a one-off edit to listing-card's price without also overwriting their saved listing profile. Three options:
   - C.1: every keystroke writes back (simple, but no one-offs possible)
   - C.2: only write back when the user clicks "Save to listing profile"
   - C.3: write back on the FIRST edit (initial population), then stop. Subsequent edits stay local to the template.

**Recommend C.1** for simplicity, matching the brand profile UX. If you want one-offs, choose C.2 and we add a "Save listing profile" button next to the property fields.

### Estimated commit count: 1-2

- **C.1:** hook file + integration in TemplateEditor + the 2 template configs. Single commit feasible. Will split into 4a (hook) + 4b (integration) if it grows past ~150 lines.

### вҡ  Open questions for sub-phase D

1. **Brand-profile UI exposure** вҖ” should the new `text` + `muted` slots be editable from `/settings` (BrandProfileForm)? Recommended: **yes**, in a single follow-up minor edit to BrandProfileForm. Otherwise users can't change the defaults from the central place.

I recommend keeping `accentColor` for backwards compat and adding a NEW slot called `onPrimary` (or `primaryTextColor`). 2 slots, distinct meanings.
3. **Per-template overrides UI placement** вҖ” render below the Brand colors section, in a collapsed accordion ("Customize colors for this template") so the form feels clean by default but power users can still drill in. Recommended.

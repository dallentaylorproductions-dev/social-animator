# H-7.14 — Render performance audit

**Date:** 2026-05-14
**Branch:** `phase-h7-14-render-perf-audit` (instrumentation Commit 1 SHA `15ce1ec`, post-v1.39.2 merge SHA `304c019`)

---

## TL;DR — what the data says

Three things in priority order:

1. **The slow MP4 exports are dominated by ffmpeg-encode** in the frame-by-frame pipeline. For Listing Flyer 5-photo Reel MP4, ffmpeg-encode is 82-85% of total export time. Tuning frame rendering would have near-zero user-visible impact; the encoder is the bottleneck.

2. **The Social Animator already uses a fundamentally faster MP4 pipeline.** Listing Flyer MP4 runs through a frame-by-frame canvas → ffmpeg.wasm encode loop on Mac Chrome desktop. Social Animator runs through a MediaRecorder + WebM→MP4 conversion path. Same machine, same browser: Listing Flyer takes 87-100s for a 240-frame video; Social Animator takes 17s for a 300-frame video. The difference is the encoder choice, not the rendering. Moving Listing Flyer and OH Promo onto Social Animator's pipeline could 4-5x their export speed essentially for free.

3. **PDF and JPEG paths are healthy across all four tools.** Everything under 2 seconds. JPEG path's main cost is the PDF→JPEG rasterization step (pdfjs), 50-60% of JPEG time across tools. Worth knowing but not a near-term priority.

The recommended H-7.15 sequence: **do W-3 (architecture consolidation) first**, then re-measure. Most of the perf wins fall out of consolidation. Don't tune the slow pipeline if we're going to replace it with a faster one.

## 1. Methodology

### Devices and configurations

- **Mac Chrome desktop** (Chrome 147 on macOS 10.15.7-equivalent) — single machine, no synthetic CPU throttling for this audit, no Mac Safari, no iPhone Safari.
- **Mid-tier Android and CPU-throttled variations were deferred** to post-launch real-customer telemetry. The single-device dogfood matrix is sufficient to identify the structural bottlenecks; device-specific tuning comes after we know what to optimize.

### Cold vs. warm definition

- **Cold** = the first MP4 export of the browser session. `ffmpeg.wasm` is not yet loaded.
- **Warm** = subsequent MP4 exports in the same session. `ffmpeg.wasm` is cached.
- PDF and JPEG paths don't have a cold/warm distinction (no `ffmpeg` involved).

### Run counts

- Listing Flyer MP4 Reel: 1 cold + 4 warm = 5 runs.
- All other paths: 1 representative run each. Single runs are not statistically robust but are sufficient to rank tools relative to each other for the audit's purpose.

### Statistical caveat

P95 with N=4 warm runs is reported as **max-of-N**, which is a conservative upper bound rather than a true P95. Real P95 numbers across populations will be available later from real-customer telemetry once the product is live.

### Test inputs

- **Photo set:** Dallen's own listing photos (mix of `Screen Shot 2025-04-06` PNGs at typical Mac retina screen-shot dimensions). The reproducible 6-photo Picsum set in `public/perf-test/` was NOT used for this audit because actual user photos were more representative.
- **Listing data:** sample property data including address, price, beds/baths/sqft, 4-5 feature lines, branded contact block.

### What was excluded

- iPhone Safari runs (would require a real device + screen recording workflow; deferred)
- Mac Safari runs (deferred; the structural bottleneck identified is engine-agnostic — same ffmpeg.wasm code path runs in any browser)
- Synthetic CPU throttling (deferred)
- Mid-tier Android (no test device available; deferred)
- Per-photo-count variation (1-photo, max-photo) — all Listing Flyer runs used 5 photos

## 2. Instrumentation harness

The `?perf=1` URL parameter activates instrumentation built into all 4 tools. When enabled:

- `performance.now()` deltas wrap canonical phases of each export path (PDF, JPEG, MP4 desktop, MP4 iOS)
- A `RunRecord` is written to `window.__perf` for each completed export
- A small dark "⚡ total time" card slides in from bottom-right after each export with the phase breakdown and a "Copy data" button
- Zero overhead when `?perf=1` is absent (instrumentation functions short-circuit to no-ops)

The harness shipped in `phase-h7-14-render-perf-audit` (Commit 1 SHA `15ce1ec`). The branch was rebuilt onto v1.39.2 as merge SHA `304c019` so the audit data reflects the current production codebase plus the bullet regression fix.

## 5. Top 3 bottlenecks ranked by P95 user-perceived impact

User-perceived = wall-clock time on an export the user clicks. Ranked by absolute time × likely frequency of use.

### #1 — Listing Flyer MP4 Reel encoding

- **P95 wall-clock:** ~100 seconds (max-of-4 warm)
- **Bottleneck phase:** `ffmpeg-encode` (84% of total time)
- **Root cause:** `ffmpeg.wasm` runs single-threaded in JavaScript with no GPU access. Frame-by-frame encoding of 240 frames is fundamentally slow in this pipeline.
- **Frequency:** Listing Flyer is the highest-traffic tool, MP4 Reel is the recommended output format.

### #2 — Open House Promo MP4 Reel encoding (same class)

- **Wall-clock:** ~34 seconds (cold, 0 photos)
- **Bottleneck phase:** `ffmpeg-encode` (82% of total time)
- **Root cause:** Same `ffmpeg.wasm` pipeline as Listing Flyer. Frame count is higher (300 vs 240) but per-frame work is lighter, netting to less total time. With realistic 4-5 photo inputs, expect this to grow to 40-60s.

### #3 — JPEG path via `pdfjs` rasterization

- **Wall-clock:** 0.5-2 seconds across tools, but consistent across all 4
- **Bottleneck phase:** `pdfjs-load` + `pdfjs-rasterize` (50-60% of JPEG time across tools)
- **Root cause:** Current architecture generates PDF first, then rasterizes the PDF page back to a canvas to produce JPEG. The double conversion is the cost.
- **Frequency:** Common operation — JPEG is the "save to Camera Roll" path users hit often.

## 6. The architectural finding — MP4 pipelines diverge across tools

This is the audit's most important takeaway.

**Listing Flyer MP4 and Open House Promo MP4** use a **frame-by-frame** pipeline:
Canvas frame render → frame captured to image → image queued → ffmpeg.wasm encodes all frames into MP4 → blob delivered

Phase names in the data: `frame-render-loop`, `frame-capture-loop`, `ffmpeg-encode`.

**Social Animator MP4** (all 10 templates) uses a **MediaRecorder + WebM→MP4 conversion** pipeline:
Canvas → MediaRecorder records live stream → WebM blob → converted to MP4 → delivered

Phase names in the data: `recorder-active`, `recorder-finalize`. **No `ffmpeg-encode` phase.**

### Why the speed difference is so large

MediaRecorder uses the browser's built-in video encoder, which is hardware-accelerated on modern hardware. `ffmpeg.wasm` is a JavaScript port running single-threaded with no GPU access. For a comparable video:

- Frame-by-frame ffmpeg: 5-10x slower than real-time playback
- MediaRecorder: roughly real-time (a 10-second video takes ~10s to record)

### Implications

If Listing Flyer's MP4 export were ported to the Social Animator's MediaRecorder pipeline:

- Wall-clock for a 5-photo Listing Flyer MP4 Reel would likely drop from ~90 seconds to **~10-15 seconds** based on a reasonable extrapolation from Social Animator's measured timings.
- Open House Promo MP4 would see a similar 3-4x speedup.
- The same consolidation eliminates the parallel code paths that caused the v1.39.2 bullet regression — same fix, two benefits.

### Trade-offs to validate during W-3

- **Quality:** MediaRecorder bitrate and codec controls differ from ffmpeg. The current ffmpeg settings (CRF 18, H.264 high profile, yuv420p) produce known-good quality. MediaRecorder may need bitrate tuning to match.
- **Timing precision:** MediaRecorder records in real-time. Frame-by-frame can in theory be faster than real-time (although in practice it's slower due to ffmpeg.wasm overhead). For 8-10 second videos this is a non-issue.
- **iOS fallback:** iOS Safari already uses MediaRecorder. Consolidating Mac Chrome to also use it means iOS and desktop run the same code path — strictly easier to maintain than today.

These are validation work for W-3, not blockers.

## 8. H-7.15 attack plan (deferred until after W-3 architectural consolidation)

**Critical sequencing:** do not start H-7.15 until W-3 is complete. The architecture consolidation buys most of the perf wins; tuning the legacy pipeline would be wasted work.

After W-3 ships, in priority order:

### H-7.15a — Re-measure post-W-3
Run the same matrix as this audit on the consolidated codebase. Validate that Listing Flyer MP4 and OH Promo MP4 dropped to the 10-15s range. If yes, move to H-7.15b. If no, investigate why MediaRecorder didn't deliver expected gains on these tools' canvas complexity.

### H-7.15b — JPEG path consolidation (if user-impact warrants)
The PDF→JPEG rasterization via pdfjs is 50-60% of JPEG time. Alternative: render JPEG directly from canvas without intermediate PDF. Investigation needed: how is the current canvas state available outside the PDF pipeline? Estimated saving: 0.5-1s on JPEG exports across all tools.

### Stop here — hard cap on perf work
The H-7.14 audit's role was to identify the top 3 bottlenecks. Beyond H-7.15a/b, additional perf tuning has rapidly diminishing returns. If users still report slowness post-consolidation, treat it as new data and run a fresh audit cycle. Don't optimize blindly.

## 9. Connection to W-1 / W-2 / W-3 (the architectural overhaul)

This audit's findings directly inform the architectural work:

- **W-1 (workflow + architecture audit):** The MediaRecorder vs ffmpeg divergence identified here is concrete evidence that "callable skills" need a shared MP4 export contract. Whichever tool calls the future "render MP4" skill should get the same fast pipeline as Social Animator.
- **W-2 (Playwright safety net):** The post-W-3 re-measurement (H-7.15a) can run automatically as part of CI once W-2 lands. Perf regressions like "an export got 2x slower" can be caught mechanically instead of via Dallen exporting and noticing.
- **W-3 (first consolidation):** The Listing Flyer MP4 → shared renderer consolidation already on the roadmap IS the perf fix. The bullet-regression-class fix and the perf fix are the same architectural change.

The audit's recommendation, in one sentence: **the right next move is W-1 → W-2 → W-3, then re-measure, NOT direct optimization of the current frame-by-frame pipeline.**

## 10. What we didn't measure (and why it's OK)

- **iPhone Safari:** Would require recording exports on a real iPhone and copying data back. The MediaRecorder pipeline ALREADY runs on iOS, so iOS likely benefits more from the consolidation than from any tuning. Banked for post-launch real-customer telemetry.
- **Mac Safari:** Same ffmpeg.wasm code path runs in any browser; the bottleneck is engine-agnostic. Confirming it on Safari would not change the recommendation.
- **CPU-throttled Mac Chrome:** Would tell us whether mid-tier hardware experiences a multiplied bottleneck. Probably yes (ffmpeg-encode is CPU-bound), but doesn't change the recommendation. Banked.
- **Per-photo-count variation:** All Listing Flyer runs used 5 photos. With more photos, frame-render-loop and frame-capture-loop both scale — but they're already <15% of total time, so doubling them still keeps ffmpeg-encode as the dominant cost. Doesn't change the recommendation.
- **Mac Safari, iPhone Safari, CPU-throttled, photo-count variations:** All banked. None would change the recommendation that the right next move is architectural consolidation, not pipeline tuning.

## 11. Confidence and known limitations

- **Single-device dataset.** All measurements from one Mac on one browser. Population P95 will be different.
- **Small N on warm runs.** N=4 max-of-N is an upper-bound estimate, not a true P95. Real P95 lands later from production telemetry.
- **One photo set per tool.** Listing Flyer used 5 photos; Open House Promo used 0 photos (placeholder template); other tools matched their typical input. Cross-tool comparisons should be read as directional, not precise.
- **One Social Animator template.** listing-carousel was the representative. Other templates may have different complexity profiles. The MediaRecorder pipeline benefit applies to all of them by construction (they all share the same export path).
- **No measurement of perceived-wait UX.** Users might tolerate 30s of export with a good progress indicator more than 10s without one. Worth keeping in mind during W-3.

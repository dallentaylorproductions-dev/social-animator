## What `timeline.render(0, ctx)` actually draws for OH Promo at t=0

OH Promo's timeline is structured differently from Listing Flyer's. There is **one track**, `id: "flyer"`, `start: 0`, `duration: totalSec`, that on every tick calls `renderFrame(ctx, t, ...)` where `t = p * totalSec`. At t=0, `renderFrame` is called.

`renderFrame` ([timeline.ts:165-206](src/tools/open-house-promo/engine/timeline.ts#L165-L206)) at t=0:

| Sub-draw | What it does at t=0 |
|---|---|
| `ctx.fillRect(0, 0, w, h)` with `state.background` | Paints page background |
| `drawHeaderBar` | **Always drawn at full opacity** ("Header bar ŌĆö always present at full opacity from t=0; only the text inside it animates") |
| `drawHeroPhoto` | `blockEntrance(0, schedule.heroPhoto=0)` ŌåÆ opacity 0 ŌåÆ returns early |
| `drawHeaderText`, `drawThumbStrip`, `drawPropertyBlock`, `drawFeaturesBlock`, `drawDescriptionBlock`, `drawAgentQrRow`, `drawFooterBar` | All check `if (e.opacity <= 0) return;` and bail at t=0 (entrance opacities are 0 for elapsed Ōēż 0) |

**Expected first-frame content: page background + colored header bar. Nothing else.**

The "full composition flash" cannot come from `renderFrame(ctx, 0, ...)` ŌĆö that function would NOT draw hero / thumbs / property / features / agent / QR at t=0.

## Hypotheses, ranked

### A ŌĆö **Pre-roll race** (most likely root cause)

`MediaRecorder` begins capturing at step 12 (`recorder.start`). The first rAF tick that paints `timeline.render(0, ctx)` doesn't fire until ~16ms later. In the gap, MediaRecorder is "open" against a freshly-cleared (transparent) canvas. iOS Safari's `captureStream` has documented quirks where:

- the first emitted frame can carry over residual content from a prior `captureStream` session on the same `<canvas>` element, OR
- the first emitted frame is the canvas state at `recorder.start` time, which on iOS may be the *previous* render's pixel buffer if the GPU-side clear from `canvas.width = X` hasn't propagated through the captureStream pipeline yet

This is consistent with the user's repro: two-format export (reel + square) reuses the same `<canvas>` element. After the reel render completes, the canvas holds the final reel composition. The square render starts, `canvas.width = 1080` is set (same value as reel) and `canvas.height = 1080` (changed from 1920). iOS Safari's GPU-backed canvas may briefly retain the prior frame in the captureStream's first-frame buffer.

**Why frame 1 of the *trimmed output* (after `ffmpeg -ss 5.5`) might still show this:** if the stale-content frame persists in the captureStream output for multiple frames at the start (before first rAF tick paints something new), all those captured frames carry the stale content. The ffmpeg seek lands at PTS=5.5, and if the captureStream's PTS clock counts the stale-content frames from t=0, then the seek-target frame might still be a stale-content one OR the boundary between stale and fresh paint.

Hypothesis A also explains why the bug is specific to **Open House Promo** and not Listing Flyer (which the user did not report seeing the flash for, per the brief): the Listing Flyer's timeline at t=0 draws only the hero photo (`hero` track, start=0, linear, opacity 1.0 from the first frame). Its first-frame "stale residue" would still be hero-only, indistinguishable from the intended frame 1. OH Promo's first frame is background + header bar only, so any stale residue carrying the FULL composition is conspicuous.

### B ŌĆö iOS Safari video-player poster artifact (player-side, not encoding)

iOS Safari's `<video>` element shows a frame from the file as a poster while loading, *before* playback begins. Some iOS Safari versions choose the *last* keyframe in the file (or the LAST frame after preload) for this poster. The user "frame 1" they're seeing may actually be the playback player's *poster* (the last animation frame) shown for ~50ŌĆō200ms before the actual frame 1 begins playing.

If this is the cause, the encoded MP4 itself is correct ŌĆö the issue is purely player-side. Verification step: would need the user to download the MP4 to the Files app and play it from there (or AirDrop to a Mac and inspect frame 1 in QuickTime). If only iOS Safari's `<video>` element shows the flash, this is the hypothesis.

### C ŌĆö `tpad` trailing clones leaking into "frame 1" via ffmpeg seek quirk (least likely)

[export.ts:319](src/engine/export.ts#L319) adds `tpad=stop_mode=clone:stop_duration=durationSec` then `-t durationSec`. If `-ss 5.5` and `-t 6` together produce an off-by-one frame at the cut boundary (e.g., the last frame before `-t` cuts is a tpad clone that bleeds into the start because of muxing reorder), the output could have the final composition at the start. Ruled out by the explicit code comment that `-ss` after `-i` is frame-accurate, but listing it for completeness.

## Proposed fix (Fix A from the brief)

**Pre-paint frame 0 to the canvas synchronously** before `MediaRecorder.start()` fires. Eliminates the pre-roll race regardless of whether the underlying iOS quirk is stale-residue or first-frame-buffer.

Concretely, in `src/tools/listing-flyer/engine/render-mp4.ts`'s `renderTimelineToWebm`, after configuring the context but BEFORE `requestAnimationFrame` + `recordCanvas`:

```ts
// PRE-PAINT: synchronously render the t=0 frame to the canvas so
// the captureStream's first emitted frame matches the intended
// frame 0 of the animation, not whatever residue was on the
// canvas from a prior render or empty captureStream buffer.
ctx.setTransform(1, 0, 0, 1, 0, 0);
ctx.fillStyle = background;
ctx.fillRect(0, 0, size.width, size.height);
timeline.render(0, ctx);
```

After the pre-paint, the rAF loop continues as before (it will redraw frame 0 again on its first tick, idempotent). The recorder.start now captures a canvas already showing the correct frame 0 content.

**Why this is the right surgical fix:**

- Single shared file (`src/tools/listing-flyer/engine/render-mp4.ts`) ŌĆö fixes both OH Promo and Listing Flyer in one edit
- No animation timing changes (durations, easing, choreography all untouched)
- No additional rAF delay before recording starts (vs. Fix B which would push recording start by 1 tick ŌĆö adds latency)
- No "discard a frame" mechanism (vs. Fix C which is messy and platform-dependent)
- Idempotent with the existing rAF loop ŌĆö first rAF tick will paint frame 0 again, which is identical to the pre-paint

**Risk:** very low. The pre-paint draws what the rAF loop would draw 16ms later anyway. The only behavior change is that the canvas is in a guaranteed-correct state at the moment `recorder.start` fires.

**Why NOT Fix B (delay recorder.start by one rAF tick):** introduces 16ŌĆō33ms additional latency to every recording. We'd need to make `recordCanvas` wait for a callback signaling "first paint done" ŌĆö adds complexity, breaks the simpler "pre-paint before scheduling" pattern.

**Why NOT Fix C (recorder.requestData / drop first frame):** leaves a dropped frame; the output would be 1 frame short of duration. Triggers `tpad` to clone-pad a frame, which means the output's first frame is now a clone of frame 1, not frame 0. Lossy and unnecessary.

# H-7.10 Input autoformat audit

Branch: `phase-h7-10-input-autoformat` · cut from main at `3b1d187` / v1.35.

This document is the gate before any input swaps. It maps every text input
across the 4 tool forms (+ the 10 Social Animator templates) to one of:

- **currency** — live `$` + thousands commas, raw digits stored
- **number** — on-blur thousands commas, raw digits stored
- **phone** — on-blur `(xxx) xxx-xxxx`, raw 10-digit string stored
- **percent** — on-blur `%` suffix, raw decimal stored
- **leave-alone** — text / small integer / free-text label / color hex / etc.

The 4 reusable components (`CurrencyInput`, `NumberInput`, `PhoneInput`,
`PercentInput`) get scaffolded in commit 2. Form-level swaps happen in
commits 3 → 6. Storage stays raw; downstream renderers untouched.

**Ambiguities flagged ⚠ — please resolve before commit 2:**

1. **`yearsExperience`** stores `"8 years"` literally. Brief said "ratios as percent". This is neither — it's a count with an embedded unit suffix. **Recommend: leave-alone.** If you want autoformat we'd need a new "number-with-suffix" component or stripping the " years" out, which changes the storage shape.
2. **`saleToListRatio`** stores `"102%"` with the % in the raw value. If we use **percent** type the storage will become `"102"` (raw decimal) and the renderer needs to append `%` at display time. **Check that downstream PresentationDocument.tsx / canvas render code reads this raw and adds `%` — or accepts a raw number and we keep the renderer's expectation unchanged.** If the render code expects `"102%"` directly, leave-alone is safer.
3. **Comp `daysOnMarket`** placeholder `"8 DOM"` — embedded label. **Leave-alone** unless you want a new field type.
4. **Comp `saleToListPercent`** placeholder `"104% S/L"` — embedded label. **Leave-alone** unless you want a new field type.
5. Brief mentioned a "commission rate '3.5' → '3.5%'" field in Listing Presentation. **No such field exists.** Closest match is `saleToListRatio` (placeholder `102%`). Did you mean that, or is the commission rate a new field to add?

## Phone autoformat — out of scope as written

The brief's smoke-test criteria mention phone formatting in Listing Flyer /
Open House Promo / Listing Presentation. **None of those tool forms expose
a phone field.** Agent phone lives in the brand profile (set once in
Settings, applied universally). Settings already has live phone
formatting via the existing `formatPhone()` helper.

If you want a `PhoneInput` component in this phase anyway (for future use
+ the FieldDef type union), we can scaffold it. But there's no callsite
in the 4 tools to swap it into. **Options:**

- A) Scaffold `PhoneInput` + add `phone` to FieldDef union, leave callsites at zero for now.
- B) Skip `PhoneInput` entirely from this phase.
- C) Expand scope to include `src/app/settings/BrandProfileForm.tsx` (it has a phone field with existing in-line formatting — would be a 1-line swap to the new component).

**Recommend C** if phone uniformity matters; otherwise A.
